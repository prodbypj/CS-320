CS 320 Portfolio Reflection
Peyton Johnston

For my CS 320 portfolio, I included the contact service files from Project One and my summary and reflections report from Project Two. These artifacts show the work I completed with software testing, automation, and quality assurance. The contact service files show how I created a basic service based on specific requirements, and the test files show how I used JUnit testing to check that the program worked correctly. The Project Two report explains my testing approach and reflects on what I learned throughout the course.

How can I ensure that my code, program, or software is functional and secure?

- I can ensure that my code is functional by carefully reviewing the requirements before writing the program and then creating tests that match those requirements. In the contact service project, this meant checking things like whether the contact ID was unique, whether fields were null, and whether the length limits    were followed. Writing unit tests helped me confirm that the code worked the way it was supposed to and helped catch errors before the program would be used by someone else.

 Security is also important because software should protect user information and prevent invalid data from being accepted. I can improve security by validating input, keeping data private when needed, and making sure the program does not allow values that could cause errors or unexpected behavior. Even though the  projects in this course were smaller, they helped me understand how important validation and testing are in larger software systems.

How do I interpret user needs and incorporate them into a program?

- I interpret user needs by breaking the requirements down into smaller parts and turning them into specific features or rules for the program. For example, in the contact service assignment, the user needed contacts to have required fields like a contact ID, first name, last name, phone number, and address. Each of those fields had rules that had to be followed, so I used those rules to guide both the code and the tests.

 This course helped me see that user needs are not always just about what the program does on the surface. They also include things like reliability, accuracy, and preventing bad data from being saved. By connecting the requirements directly to the code and test cases, I can make sure the final program actually matches what the user asked for.

How do I approach designing software?

- When I design software, I try to start with the requirements and think about the main purpose of the program first. From there, I break the program into smaller parts so that each class or method has a clear job. For the contact, task, and appointment services, this meant separating the object classes from the service classes and then writing tests for each part.

 I also learned that testing should be part of the design process, not something that only happens at the end. Thinking about how the code will be tested helped me write cleaner and more organized code. My approach to designing software is to keep the program simple, make sure each part connects back to a requirement, and use testing to confirm that the program works correctly.
